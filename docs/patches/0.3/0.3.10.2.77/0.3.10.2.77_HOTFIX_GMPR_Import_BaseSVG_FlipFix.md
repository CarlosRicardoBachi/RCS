# 0.3.10.2.77 — HOTFIX GMPR Import: Base SVG visible + Flip fix

## Problema (P0)
1) Importar GMPR crasheaba con:
   - `Transform.__init__() got an unexpected keyword argument 'flip_x'`
2) Aunque el GMPR traía `svg_embedded`, el SVG **no aparecía** en escena (solo entraba el raster o quedaba “vacío”).

## Causa raíz
- El conversor de transform GMPR→RCS usaba campos que **no existen** en `Transform` (`flip_x`, `opacity`, `lock_aspect`).
- El import GMPR materializaba el SVG embebido a archivo temporal, pero **no creaba** el `SceneObject` de tipo `svg`, por lo que nunca se renderizaba en el canvas.

## Fix (qué cambia)
- `rcs/core/gmpr_io.py`
  - `_gmpr_raster_transform_to_rcs_transform()` ahora mapea flips correctamente:
    - GMPR `flip_h/flip_v` (nuevo) o `flip_x/flip_y` (legacy) → RCS `flip_h/flip_v`.
    - Ignora keys extra (tolerante).
  - `_update_gmpr_raster_transform_dict()` escribe `x,y,sx,sy,rot,flip_h,flip_v` y mantiene legacy keys si existen.
  - `gmpr_to_project()` ahora agrega un objeto base:
    - `SceneObject(type=svg, id="SVG_BASE", source=<svg_embedded materializado>)`
    - Esto hace que el SVG embebido sea **visible** y usable como referencia de alineación.
- `rcs/core/version.py`
  - bump a `0.3.10.2.77`

## Checklist de prueba (5 minutos)
1) Abrir RCS y hacer **Archivo → Importar GMPR** sobre `proyectoabc.GMPR`.
2) Confirmar que aparecen **2 objetos**:
   - `SVG_BASE` (tipo svg) y el raster.
3) Confirmar que el import **no crashea** (no hay error de `flip_x`).
4) Mover/zoom/pan: el SVG se ve y el raster está alineable visualmente.
5) Guardar GMPR (si aplica tu flujo) y reabrir para validar que no rompe.

## Notas
- Este hotfix es deliberadamente mínimo: no cambia el pipeline de import SVG “normal” desde biblioteca.
- Si el GMPR no trae `svg_embedded` y tampoco trae un `svg_path` válido, se loguea warning y no se crea `SVG_BASE`.
